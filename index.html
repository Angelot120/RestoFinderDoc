Documentation du projet "RestoFinder"

Objectif du projet


L'application "RestoFinder" a pour but de permettre aux utilisateurs de consulter une liste de restaurants à proximité et de commander des plats à emporter. Les principales fonctionnalités de l'application incluent :


  • Afficher la liste des restaurants à proximité
  • Afficher les détails d'un restaurant (nom, adresse, horaires, menu, etc.)
  • Ajouter des plats au panier et passer une commande

Fonctionnalités principales


  1. Affichage des restaurants à proximité : L'application utilise Geolocator pour obtenir la position actuelle de l'utilisateur et récupère les restaurants dans un rayon de 5 km via l'API.
  2. Détails des restaurants : En cliquant sur un restaurant dans la liste, l'utilisateur peut accéder à une page avec les détails complets du restaurant : adresse, description, horaires, menu, etc.
  3. Gestion du panier : L'utilisateur peut ajouter des plats à son panier, et passer une commande.

Architecture - Clean Architecture


L'application Flutter utilise une architecture propre, connue sous le nom de Clean Architecture, qui sépare clairement les préoccupations en plusieurs couches indépendantes et bien définies. Cette approche vise à rendre le code plus testable, maintenable et évolutif, tout en favorisant la séparation des responsabilités.


Structure des Couches


  • Domain Layer (Couche Domaine) :

    Cette couche contient la logique métier de l'application. Elle est indépendante de toute bibliothèque ou technologie spécifique et ne dépend d'aucune autre couche. Elle inclut les entités, les cas d’utilisation (use cases) et les interfaces des repositories.

  • Data Layer (Couche Données) :

    Cette couche est responsable de la gestion des données. Elle contient les implémentations des repositories, des sources de données (API, bases de données locales, etc.) et la sérialisation des données. Cette couche est dépendante de la couche Domaine et fait appel à ses interfaces pour l'accès aux données.

  • Presentation Layer (Couche Présentation) :

    La couche présentation gère l'interface utilisateur (UI). Elle contient les widgets, les contrôleurs d'état (State Management), et les ViewModels qui sont responsables de la logique de présentation. Cette couche dépend des cas d'utilisation du domaine et interagit avec la couche Data pour afficher les données.

  • Dependency Injection (Injection de Dépendances) :

    Un service d'injection de dépendances est utilisé pour fournir les objets nécessaires à chaque couche, en s'assurant que les dépendances sont correctement instanciées et injectées dans les classes appropriées.


Avantages de Clean Architecture


  • Testabilité : Chaque couche est isolée et indépendante des autres, ce qui facilite les tests unitaires et l'intégration.
  • Indépendance des technologies : Les détails de l'implémentation (par exemple, le framework Flutter, la base de données) sont isolés, permettant de remplacer ou de modifier ces technologies sans affecter le reste de l'application.
  • Évolutivité : Le code peut être facilement étendu avec de nouvelles fonctionnalités ou modifié sans perturber l'ensemble du système.
  • Maintenance facilitée : Le code est organisé de manière à ce que les modifications dans une couche n'affectent pas les autres couches, réduisant ainsi le risque de régressions.

Utilisation du Package get_it pour l'Injection de Dépendances


Dans cette application, le package get_it est utilisé pour gérer l'injection de dépendances. Il permet de décorréler les différentes couches et de garantir que chaque classe obtient ses dépendances sans les créer directement. Cela améliore la flexibilité et facilite les tests.
